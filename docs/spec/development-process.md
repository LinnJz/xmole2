# 软件生命周期与开发流程规范

## 1. 信息基线

需求、官方格式标准、spec、ADR、测试和实现构成同一条可追踪链。`docs/office-standard/` 中对应版本的官方 PDF 约束文件格式事实，其 Markdown/OCR 派生产物用于检索与复核；具体使用和冲突处理见 `office-standard.md`。聊天、口头约定和参考项目行为不构成长期基线；重要结论必须进入仓库文档。

设计会议必须在 `docs/meetings/` 保存完整会话事实源以及逐项决议、状态与规范落点。完整会话的用户确认内容优先于摘要；摘要 spec 和决议索引不能替代事实源，也不得加入未经确认的具体方案。每个重大条款必须可追溯到完整会话或 ADR。

项目架构与公共契约的优先级为：已接受 ADR 与最新 spec → 明确验收测试 → 实现。文件格式事实以匹配版本的官方标准为准；标准、spec 与测试不一致时必须先解决冲突。

## 2. 变更流程

每项功能或修复按以下顺序进行：

1. **需求定义**：描述用户场景、输入输出、格式范围、保真与安全要求。
2. **影响分析**：识别涉及 target、公共 API、dialect、资源预算、fixture 和互操作方。
3. **设计决策**：跨模块依赖、公共契约或长期方向变化必须先写 ADR。
4. **规范更新**：同步修改相关 spec，并给出可验证的 acceptance criteria。
5. **实施计划**：拆成可独立验证的垂直切片，禁止以“先写完再补测试”为计划。
6. **测试先行**：先建立 contract/negative/round-trip 证据；缺陷必须有回归测试。
7. **实现**：只修改已授权范围，遵守 target 依赖和 public/private 边界。
8. **本地验证**：格式化、配置、构建、测试、必要 sanitizer/benchmark。
9. **评审**：同时检查规范符合性、正确性、安全、保真、性能和许可证 provenance。
10. **合入与发布**：CI 全绿，文档与版本记录完整，发布制品可复现。

## 3. 变更分类

- Patch：不改变公共契约的修复与内部优化；需要回归测试。
- Minor：向后兼容的能力或 API 增加；需要 spec/能力矩阵更新。
- Breaking：公共语义、依赖方向、文件保真承诺或 ABI 改变；必须 ADR、迁移说明和版本策略。
- Security：独立威胁分析、修复窗口、负向测试和披露记录。

## 4. Definition of Ready

开始编码前必须具备：明确范围、目标格式/dialect、对应官方标准及版本、输入安全假设、保真策略、错误/诊断、资源预算、验收标准和合法 fixture。缺少关键决策或标准依据时不得以临时代码代替设计。

涉及第三方库时，还必须按 `dependencies.md` 确认固定版本的实际接口、imported target、许可证和允许依赖层；不得依据上游 main 或模型记忆直接编码。

## 5. Definition of Done

功能只有在以下全部满足时完成：

- 实现遵循 target 边界且 public API 无第三方泄漏；
- unit、contract、negative、必要 round-trip/interop 测试通过；
- 保存/转换产生正确报告，未知内容策略有证据；
- 预算和取消路径可测试，失败不留下部分输出；
- 文档、能力矩阵、ADR 和 changelog 已更新；
- clang-format、Windows CI 及当前要求的平台任务通过；
- 性能敏感变更具有同 fixture 的前后数据；
- 引入/借鉴代码完成许可证与 provenance 检查。

## 6. 评审与职责

架构评审关注依赖方向和长期契约；格式评审关注规范条款与互操作；安全评审关注不可信输入和资源边界；代码评审不能替代任何一类专项证据。

任何 reviewer 都可以因规范缺失、fixture 来源不明、静默保真损失或越层依赖阻止合入。

## 7. 发布与弃用

1.0 前允许受控的源码 API 调整，但每次 breaking change 都提供迁移说明。弃用 API 先标记、给出替代方案和移除版本，再按版本策略删除。

发布必须包含组件清单、依赖锁定信息、平台/编译器矩阵、能力矩阵、已知保真限制、安全说明和校验哈希。

## 8. 缺陷与回溯

生产缺陷先建立最小复现和影响范围，再修复根因并补回归测试。涉及数据丢失、安全或错误格式写出的事件必须形成 postmortem，并评估是否需要 ADR/spec 调整。
