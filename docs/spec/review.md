# 代码审查规范

状态：Normative Baseline  
版本：0.1  
适用范围：xmole2 模块级的实现审查、重构审查和第三方代码评估

## 1. 目标

审查是正式质量活动，不替代测试。审查回答"实现是否符合规范、是否有隐藏缺陷、是否有性能或可维护性债务"，而非"它能不能编译运行"。

每次审查必须输出可操作的发现列表，按严重性分级，并明确指出当前实现与对应 spec 之间的一致或偏差。

## 2. 审查层次

### 2.1 架构对齐度

- 模块依赖方向是否与 `architecture.md` 一致，是否存在越层或反向依赖。
- 第三方类型是否泄漏到 public header（allocation、pugixml、minizip-ng、fmt、frozen、OpenSSL 等）。
- public API 是否只使用 xmole2 自有类型和标准库类型。
- 模块在 CMake target 拓扑中的位置是否符合其职责描述。

### 2.2 正确性

- 线程安全假设与实现是否一致（atomic 的 memory order、共享可变状态的保护）。
- 整数溢出、符号转换、窄化赋值是否被防御或通过证明无害。
- 资源清理是否是异常安全的（RAII、析构函数不丢错误、close/flush 顺序）。
- 外部输入验证是否覆盖路径遍历、空字节、超长字段、不可信偏移。
- 错误链（cause/native_code）是否在模块边界完整传递。

### 2.3 性能

- 系统调用的频率：`size()`、`seek()`、`stat` 是否可缓存，是否在循环中重复调用。
- I/O 模式是否匹配操作系统原语（`pread`/`OVERLAPPED` 的恰当使用，同步 vs 异步）。
- 冗余解析或重复构造：例如 ZIP 中央目录是否被反复解析、BackendArchive 是否可复用。
- 堆分配频率：error 构造、shared_ptr 复制、`std::string` 临时对象在热路径上的比重。
- 过早抽象的间接性：不必要的 unique_ptr 封装、virtual 分派、删除的 move 操作强制堆分配。

### 2.4 设计合理性

- 抽象边界是否正确：模块是否知道它不应该知道的语义。
- 类型体积与传参方式：大对象是否值传递、是否应使用 shared_ptr 或 span。
- API 的误用阻力：是否有防止调用者弄错生命周期、所有权、线程假设的措施。
- 扩展点是否必要且稳定：前向声明但无实现的占位符应有 TODO 或自明理由。

### 2.5 测试覆盖

- 每个 public API 是否有 contract test 覆盖正常路径和预期错误。
- 负向测试：损坏输入、恶意输入、预算耗尽、取消、故障 source。
- 并发安全是否被测试（或明确声明不保证并在线程安全边界测试）。
- 大文件/边界条件（>4GB、INT32_MAX、size_t 溢出边界）是否被测试。

### 2.6 代码质量

- 命名、格式化、文件布局是否与 `code-style.md` 一致。
- 跨平台分支是否正确：`_WIN32`/`else` 两端都有实现，非存根。
- magic literal 是否应提升为 `constexpr` 常量。
- 析构函数和 close 路径的错误忽略是否一致且有正当理由。

## 3. 审查报告模板

审查报告必须使用以下六节结构：

```
## 1. 架构对齐度

## 2. 正确性

## 3. 性能

## 4. 设计合理性

## 5. 测试覆盖

## 6. 代码质量细节
```

每节内部按严重性降序排列：

- **A. 正确性问题** — 可能导致数据损坏、安全漏洞或未定义行为
- **B. 性能问题** — 具有可测量的实际影响（系统调用、多余 I/O、O(n) 误为 O(1)）
- **C. 设计问题** — 抽象泄漏、可维护性债务、不必要的扩展点
- **D. 微问题** — 无实际影响但应修复的不一致或风格问题

每项必须标注文件路径和行号。

## 4. 审查者守则

1. **假定实现可能有错**。不以"显然是正确的"跳过验证。
2. **先验证据优先于模型推理**。对性能可测量、对安全性可证明、对正确性可测试。
3. **问为什么**。识别重复模式中的差异（例如一个析构函数 flush、另一个不 flush）。
4. **报告问题不分先后**。关键问题先于风格问题，但不因此跳过风格审查。
5. **结构不只在口头上**。所有发现必须有可重现的文件+行号引用。
6. **不确定时问而不是猜**。标记需要补充信息的项目并说明需要什么证据。
7. **不奉承**。确实优秀的实现用精确描述代替夸奖。"可移动对象没有禁用 move，减少了堆间接性"比"设计很好"更有用。

## 5. 与开发流程的关系

审查是 `development-process.md` 第 2 节 第 9 步"评审"的活动指南。它既可以作为 CI 前的手工评审 checklist，也可以作为 AI agent 审查 skill 的行为基线。

每次审查后，发现应至少导致以下之一：
- 修复被审查代码
- 新增或更新 contract test
- 更新对应 spec 或 ADR
- 创建技术债务跟踪项

审查不产生新的规范。规范变更走标准 ADR 流程。
