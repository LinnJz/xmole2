<task-instruction>

执行一次纵向对比审查：将 xmole2 模块实现与 `references/` 下同领域第三方开源库进行交叉对照，识别正确性问题、可吸收的架构设计、性能优化机会和功能扩展点，输出结构化分析报告。

## 执行前

1. **读取项目相关规范**: 读取 `docs/spec/` 中与被审查模块直接关联的规范文档（architecture.md、公共 API、安全、IO、测试等），确认模块 target 定义和职责边界。
2. **读取项目相关 ADR**: 检查 `docs/adr/` 中影响本模块的已有决策。
3. **读取项目实现**: 读取被审查模块所在目录的全部 `.hpp`、`.cpp` 和 `CMakeLists.txt`。
4. **读取 contract test**: 读取 `tests/` 下对应模块的 contract test，了解测试覆盖路径和遗漏边界。
5. **读取 reference-snapshots.md**: 读取 `docs/reference-snapshots.md`，确认该模块关联了哪些第三方参考项目及其研究范围。
6. **读取参考实现**: 根据 `docs/reference-snapshots.md` 的指引，逐个读取 `references/` 下关联的第三方库源码。聚焦以下维度：
   - 模块边界和抽象层次
   - 关键算法实现（如 DIFAT/FAT 链遍历、MiniFAT 映射、目录树验证）
   - 错误处理模式
   - 资源管理和生命周期
   - 性能敏感路径
   - 测试覆盖策略

## 审查者守则

- **假定实现可能有错**。不以"显然正确"跳过验证，对每一处差异需对照规范确认。
- **先定规范再定对错**。与 MS-CFB 等官方标准冲突时以标准为准，与参考实现冲突时以规范为准，参考实现自身的 bug 需标注。
- **每项发现必须标注文件路径和行号**（项目源码和参考源码均需标注）。
- **引用有 bug 的参考实现时要明确标注**，避免误导。
- **不确定时主动索要补充信息**。标记需要什么样的证据后再继续。
- **不奉承**。确实优秀的实现用精确描述代替夸奖；评价应可验证。
- **记住项目正处于开发阶段**，代码实现可能有疏漏，反复推敲每一项判断。

</task-instruction>

<review-dimensions>

按以下五个维度组织分析，每个维度内按严重性降序排列：

### A — A 正确性问题
数据损坏、安全漏洞、未定义行为、违反规范明确约束。必须修复。
标注格式：`项目源码:行号` → 问题描述 → 规范依据 → 参考实现有无同样问题 → 修复方向

### B — 架构可借鉴点
参考实现中架构优于当前实现的设计。按引入优先级排列：
- 设计：抽象层次、模块边界、API 设计、生命周期管理
- 参考实现位置：`参考路径:行号`
- 引入成本：低/中/高
- 建议：具体如何引入

### C — 性能优化
可测量的实际性能差异。标注参考实现的具体算法策略和当前实现的差异：
- 参考路径:行号（参考实现的关键性能路径）
- 项目路径:行号（当前实现的对应路径）
- 差异描述：算法复杂度、I/O 模式、内存分配模式
- 注意：参考实现若有 bug 或过度优化需标注

### D — 功能差距
参考实现具备但当前实现缺失的功能。按对上游消费方的需求紧急度排列：
- 功能描述 + 参考路径
- 适用场景
- 依赖的上游模块 / 配合组件
- 建议实现优先级

### E — 代码质量问题
- 重复代码（参考实现中无重复 vs 当前实现多处重复）
- 命名不一致
- magic number
- 注释缺失或过时
- 测试覆盖遗漏

</review-dimensions>
